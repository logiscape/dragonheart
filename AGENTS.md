# AGENTS.md

This file provides guidance to Codex and Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Dragon Heart is a Windows desktop app (Tauri v2 + React 18 + TypeScript) for persistent, character-driven conversation, powered entirely locally by Ollama + Gemma 4. The product metaphor is a *relationship* with a character who has a stable interior life (the **Soul Document**) and remembers you between sessions — not a productivity chat surface. The repo implements Phases 0–2 of `Dragon_Heart_Concept_Plan.md` (read it for the design intent behind the engine).

## Commands

```bash
npm install
npm run tauri dev        # launch the desktop app (Vite + Rust shell)
npm run dev              # Vite frontend only (no Tauri shell)
npm test                 # engine unit tests (vitest run)
npm run test:watch       # vitest watch
npm run typecheck        # tsc --noEmit (app code; excludes tests)
npm run build            # typecheck + Vite bundle
npm run tauri build      # produce the desktop bundle

# single test file / test name
npx vitest run src/engine/__tests__/memory.test.ts
npx vitest run -t "rolls up"

# live engine ↔ Ollama integration tests (require a running Ollama)
DH_LIVE=1 NODE_OPTIONS=--experimental-sqlite npx vitest run live
```

There is no linter; correctness is enforced by `tsc` (strict, plus `noUnusedLocals`/`noUnusedParameters`/`noUncheckedIndexedAccess`) and vitest.

## Architecture

The codebase is split so that **all logic lives in a pure, UI-free, dependency-injected engine**, and everything else is a thin shell around it.

```
src/engine/      pure TypeScript — NO UI, NO Tauri imports. Ollama client,
                 layered context assembler, memory, lorebook, repositories,
                 character cards, persona probes. Unit-tested in __tests__/.
src/engine/ports.ts   the ONLY way the engine touches the outside: Db,
                      OllamaTransport, Clock interfaces (+ newId()).
src/adapters/    wire the ports to Tauri: tauriDb / tauriOllama / tauriFiles.
src/state/store.ts    app store (useSyncExternalStore).
src/ui/ds/       design-system components; src/ui/screens/ the actual screens.
src-tauri/src/   thin Rust shell: ollama.rs (reqwest proxy streaming Ollama
                 NDJSON to the webview over a tauri::ipc::Channel — avoids CORS
                 + gives token streaming), files.rs, lib.rs. SQLite via
                 tauri-plugin-sql.
```

`src/engine/index.ts` (`class Engine`) is the facade the UI talks to — it owns the turn loop (`send()`) and wires repositories + Ollama client + context assembler + memory/lore. It's constructed via `Engine.create(db, transport, clock)`, so tests inject in-process fakes and the desktop shell injects Tauri ports. Keep the engine free of UI/Tauri imports — that boundary is the whole point (it lets the engine later back a web surface; plan §3).

### Two systems to understand before editing the engine

**The Layered Context Model** (`src/engine/context.ts`, `assembleContext`). Every turn, the model's context is composed from independent layers — Soul Document → user/persona → relationship → triggered lore + recalled memory → scene → recent verbatim history → control tokens — then fit to a managed `num_ctx` budget. This is what prevents the "personality goes flat by message 30" failure. Changes to prompt composition belong here, layer by layer.

**Two-tier memory** (`src/engine/memory.ts`, orchestrated in `Engine.send`/`maybeRollup`). Recent turns are kept verbatim; once history exceeds `rollupThresholdTokens`, older turns (beyond `recentVerbatimTurns`) are summarized by the local model into durable `Memory` rows and marked `summarized`. Memories are recalled per-turn semantically (embeddings) or by keyword (`rankMemoriesForRecall`). Rollup runs in the background after a reply and never blocks it. Memory extraction only *proposes* — the user curates everything in the Studio; a wrong memory is worse than a forgotten one.

## Critical conventions & gotchas

- **`num_ctx` must always be set explicitly** (default 16K). Ollama silently defaults Gemma to 4K, which is fatal for a memory-driven app.
- **Embeddings need a dedicated model.** Chat models (incl. `embeddinggemma`) return 501 on `/api/embed`; only models like `nomic-embed-text` work. Semantic recall degrades gracefully (`Engine.embed` returns `null`) when off/unavailable.
- **SQL uses `?` positional placeholders** in the engine. `tauriDb` translates `?` → `$1,$2…` for SQLx; the `node:sqlite` test adapter uses `?` directly. TEXT UUID keys are generated by callers via `newId()`.
- **JSX is the classic runtime** (`tsconfig jsx:"react"` + `react({jsxRuntime:"classic"})`), so components keep `import React`.
- **Tests have their own tsconfig.** App `tsconfig.json` sets `types:[]` and excludes `__tests__/`; `tsconfig.test.json` adds `@types/node`. The build typecheck would otherwise choke on the Node-based `live.test.ts`.
- **Live tests** use `node:sqlite` via a runtime `createRequire` (so Vite doesn't pre-resolve it) and require Node 22's `--experimental-sqlite` flag; they only run with `DH_LIVE=1`.
- **Windows native-deps quirk:** if Vite/Tauri CLI fail with MODULE_NOT_FOUND for a native `.node`, manually install the matching `@rollup/rollup-win32-x64-msvc` / `@tauri-apps/cli-win32-x64-msvc` at a version matching the parent package (npm optional-deps bug).
- **Path aliases** `@engine`, `@ui`, `@adapters`, `@state` (and `@assets`) are defined in `tsconfig.json` and mirrored in `vite.config.ts` / `vitest.config.ts` — update all when adding one.
- `.design-system/` is extracted (gitignored) reference from the committed `Dragon_Heart_Design_System.zip`; the "candlelit hearthside" tokens live in `src/styles/tokens/`. Fonts load from Google Fonts and degrade to system serifs offline.
